01

הקדמה ותכנון האסטרטגיה

לפני שנוגעים בקוד — מבינים מה אנחנו עושים, ולמה.

שתי אסטרטגיות עיקריות להעברה

קיימות שתי גישות עיקריות להעברה מ-ClearCase ל-Git/GitLab. המדריך הזה מתמקד באסטרטגיה השנייה — העברת Snapshot (הגרסה האחרונה בלבד) — שהיא הבחירה שלך לפי התיאור.

גישה א' — העברת היסטוריה מלאה

שימוש בכלים כמו git-cc, ClearCase Exporter, או סקריפטים מותאמים להמרת היסטוריית ה-versions. מורכבת, איטית, ונוטה לשגיאות עם סוגי קבצים בינאריים וענפים מורכבים.

גישה ב' — Snapshot (המסלול שלנו) ✓

לוקחים את הגרסה האחרונה מהסיפרייה המקומית, מבצעים ניקוי יסודי, יוצרים .gitignore מדויק, ודוחפים ל-GitLab כ-commit ראשון. פשוט, מהיר, ונקי.

i
למה Snapshot מספיק ברוב המקרים?

היסטוריית ClearCase מישארת ב-ClearCase עצמו (לקריאה בלבד, לצרכי ארכיון). ב-GitLab מתחילים "דף חדש" ונקי. אם צריך פעם לבדוק מה היה בעבר — הארכיון הישן זמין.

02

דרישות מקדימות

הכלים והזיהויים שצריך להכין לפני שמתחילים.

כלים חיוניים

  1. Git for Windows — הגרסה העדכנית מ-git-scm.com. כוללת Git Bash, שהוא הטרמינל שבו נשתמש לכל הפקודות במדריך.
  2. Git Credential Manager — מגיע בדרך כלל עם Git for Windows. מנהל את ה-credentials מול GitLab.
  3. חשבון GitLab — עם הרשאה ליצירת Projects בקבוצה הרלוונטית (Group/Subgroup).
  4. עורך טקסט טוב — VS Code, Notepad++ או דומה, לצפייה ועריכה של קבצי .gitignore וקבצי קונפיגורציה.
  5. Visual Studio — הגרסה/גרסאות של הפרויקטים (2010, 2015, 2019, 2022), לוידוא build אחרי ההעברה.

הגדרות Git גלובליות — פעם אחת, לכל החיים

הפקודות הבאות מגדירות זהות גלובלית, line endings מתאימים ל-Windows, ודיפולטים שחוסכים הרבה כאב ראש:

BASH · Git config
# זהות המשתמש — חשוב שתתאם לזו של GitLab
git config --global user.name "Amit Cohen"
git config --global user.email "amit.cohen@company.co.il"

# טיפול נכון ב-line endings בסביבת Windows
git config --global core.autocrlf true

# שם ברירת מחדל לענף הראשי
git config --global init.defaultBranch main

# מונע קבצים גדולים בטעות (אזהרה מעל 50MB, שגיאה מעל 100MB)
git config --global core.bigFileThreshold 50m

# rebase as default for pull (אופציונלי — תלוי זרימת העבודה)
git config --global pull.rebase false

# תמיכה בשמות קבצים ארוכים (חשוב! בעיקר לפרויקטי .NET)
git config --global core.longpaths true

# בדיקה
git config --global --list
!
שים לב ל-core.autocrlf=true

זו ההגדרה הנכונה ל-Windows: Git יחזיק את הקבצים ב-repo עם LF, אבל יחליף ל-CRLF ב-checkout. בלעדיה, Visual Studio עשוי לסמן לך את כל הקבצים כ"שונו" אחרי clone.

03

הכנה וניקוי הפרויקט

לפני שמעבירים — מסירים את כל מה שלא שייך ל-repo.

שלב 1: העתקה לתיקיית Staging

BASH · Git Bash
# פתח Git Bash ונווט לסיפריית העבודה
cd /c/Migration

# יצירת תיקיית staging אם לא קיימת
mkdir -p _staging

# העתקה של הפרויקט (דוגמה ל-VS2019)
cp -r _sources/ProjectGamma-VS2019 _staging/ProjectGamma

# מעבר לתיקיית הפרויקט המועתקת
cd _staging/ProjectGamma

שלב 2: הסרת קבצי ClearCase

ClearCase מוסיף קבצים ותיקיות משלו שלא צריכים להגיע ל-Git. הפקודות הבאות מסירים אותם מהעותק ב-staging:

BASH · Clean ClearCase artifacts
# יש לוודא שאתה בתוך תיקיית הפרויקט ב-_staging!
pwd   # יציג: /c/Migration/_staging/ProjectGamma

# מחיקת תיקיות פנימיות של ClearCase
find . -type d -name "lost+found" -exec rm -rf {} +
find . -type d -name ".cc_data" -exec rm -rf {} +

# מחיקת קבצי checkout/view של ClearCase
find . -name "*.keep" -delete
find . -name "*.keep.*" -delete
find . -name "*.contrib" -delete
find . -name "*.contrib.*" -delete
find . -name "view.dat" -delete

# מחיקת תיקיית .copyarea.db (אם קיימת)
find . -name ".copyarea.*" -delete
i
הקבצים האופייניים של ClearCase

.keep נוצרים כשעושים hijack, .contrib נוצרים כשיש merge conflict, ו-lost+found היא תיקיית מטא-דאטה פנימית. אף אחד מהם לא רלוונטי ל-Git.

שלב 3: ניקוי Build Artifacts

אלה המחשולים הגדולים — תיקיות bin, obj, packages, .vs — שיכולות להכפיל את גודל הפרויקט פי 10. חשוב למחוק אותן לפני ה-commit הראשון.

BASH · Clean build artifacts
# תיקיות build של Visual Studio (כל הגרסאות)
find . -type d -name "bin" -exec rm -rf {} +
find . -type d -name "obj" -exec rm -rf {} +
find . -type d -name "Debug" -exec rm -rf {} +
find . -type d -name "Release" -exec rm -rf {} +
find . -type d -name "x64" -exec rm -rf {} +
find . -type d -name "x86" -exec rm -rf {} +

# תיקיית .vs (VS 2015+) — מכילה IntelliSense cache, אין צורך ב-Git
find . -type d -name ".vs" -exec rm -rf {} +

# NuGet packages — יורדים מחדש מה-feed, אין צורך ב-Git
find . -type d -name "packages" -exec rm -rf {} +

# node_modules (אם יש חלק front-end)
find . -type d -name "node_modules" -exec rm -rf {} +

# קבצי משתמש של Visual Studio
find . -name "*.suo" -delete
find . -name "*.user" -delete
find . -name "*.userosscache" -delete
find . -name "*.sln.docstates" -delete

# קבצי ReSharper (אם בשימוש)
find . -name "*.DotSettings.user" -delete
find . -type d -name "_ReSharper*" -exec rm -rf {} +

# לוידוא — בדיקת הגודל הסופי
du -sh .
!
זהירות עם תיקיות בשם Debug/Release

אם הפרויקט מכיל תיקיות מקור עם שמות כאלה (נדיר, אבל קורה) — אל תריץ את הפקודות באופן עיוור. בדוק עם find . -type d -name "Debug" תחילה, וראה מה נמחק.

שלב 4: זיהוי קבצים שכדאי לשקול

סרוק לאיתור קבצים גדולים או מסוג שלא רצוי לכלול ב-repo:

BASH · Audit
# קבצים גדולים מ-10MB
find . -type f -size +10M -exec ls -lh {} \; | awk '{print $5, $9}'

# קבצי DLL/EXE (ייתכן שאלה חבילות צד שלישי שצריכות להישאר, או artifacts למחיקה)
find . -name "*.dll" -o -name "*.exe" -o -name "*.pdb"

# סה"כ קבצים — שגודל ה-repo לא יהיה מפלצתי
find . -type f | wc -l

# גודל כולל
du -sh .
!
כלל אצבע לגבי גודל

Repo בריא ב-GitLab הוא מתחת ל-1GB. אם אתה רואה יותר — משהו לא נקי. קבצים בודדים מעל 100MB דורשים Git LFS (Large File Storage) או הוצאה מה-repo.

איך לזהות את גרסת הפרויקט?

פתח את קובץ ה-.sln בעורך טקסט. השורה הראשונה היא:

Microsoft Visual Studio Solution File, Format Version XX

המספר XX מציין את הגרסה לפי הטבלה הבאה:

  • Format Version 11 — Visual Studio 2010
  • Format Version 12 — Visual Studio 2015 / 2019
  • Format Version 12.00 + VisualStudioVersion=17 — Visual Studio 2022

04B

פרויקטי Web — VS Code / Angular / TypeScript

תבנית gitignore ייעודית לפרויקטי front-end מודרניים, עם דגש על Angular.

למה צריך gitignore אחר?

פרויקטי Web מבוססי Node.js / Angular / React הם עולם אחר לחלוטין מ-.NET:

node_modules/

תיקייה ענקית (מאות MB עד GB) שנוצרת מחדש מ-package.json בכל npm install. לעולם לא ל-Git!

dist/ ו-build/

תוצרי בנייה (compiled bundles, minified JS, source maps). מתחדשים בכל build, אין צורך לשמר אותם.

.env files

מכילים סודות (API keys, DB passwords). חייבים להישאר מחוץ ל-repo. רק .env.example נכנס.

.angular/cache

cache של Angular CLI לזירוז builds. גודל יכול להגיע ל-GB. מקומי בלבד.

הכנת GitLab

יצירת Project ריק ב-GitLab שיקבל את הקוד.

יצירת Project חדש

  1. התחבר ל-GitLab (בין אם בענן ב-gitlab.com או על שרת פנימי/On-Premise).
  2. לחץ על "New project" → "Create blank project".
  3. מלא את הפרטים:
    • Project name: שם ברור, למשל project-gamma
    • Project URL: בחר את הקבוצה (Group) הנכונה
    • Visibility Level: Private / Internal / Public לפי המדיניות
    • Project Configuration: בטל את "Initialize repository with a README" — אנחנו מביאים משלנו
  4. לחץ "Create project".
  5. העתק את ה-Clone URL (HTTPS) מהמסך שיפתח — נצטרך אותו בשלב ה-push.
!
בלי Initialize!

אם GitLab ייצור README אוטומטי, ה-remote כבר לא יהיה "ריק" ונצטרך לעשות merge מורכב. תמיד צור פרויקט ריק לחלוטין כשמעבירים קוד קיים.

אימות — HTTPS או SSH?

HTTPS + Personal Access Token

הכי פשוט ל-Windows. בפעם הראשונה יישמרו ה-credentials על ידי Git Credential Manager.

יצירת Token: ב-GitLab — User Settings → Access Tokens → הרשאות read_repository + write_repository.

SSH Key

בטוח יותר ואין צורך בהקלדת סיסמה. דורש יצירת מפתח והוספתו ל-GitLab.

יצירת SSH key (אם בחרת SSH)

BASH · SSH key

ssh-keygen -t ed25519 -C "amit.cohen@company.co.il"

# הצגת המפתח הציבורי — תעתיק אותו ל-GitLab
cat ~/.ssh/id_ed25519.pub

# ב-GitLab: User Settings → SSH Keys → הדבק והוסף

# בדיקת חיבור
ssh -T git@gitlab.com

שלב 2: בדיקה ראשונה — מה ייכלל ב-commit?

לפני ה-git add, כדאי לוודא שה-.gitignore עובד כמו שצריך:

# רשימת קבצים שנקלטים בגלל .gitignore (אם רצית לוודא שמשהו לא נכנס)

הודעת commit מושקעת — למה?

זה ה-commit היחיד עם "היסטוריה" במובן מסוים. לרשום כאן מאיפה הגיע הקוד, מאיזו גרסת VS, ומתי — יעזור למפתחים שיצטרפו בעתיד להבין את ההקשר.

שלב 4: חיבור ל-remote של GitLab

BASH · Add remote & push

# דחיפה ראשונה — יוצרת את ה-branch main ב-GitLab ומקשרת אליו

הדגל -u

-u origin main קובע את ה-upstream לאותו branch, כך שבפעמים הבאות מספיק git push ללא פרמטרים. פעם אחת בלבד!

טעויות נפוצות ופתרונות

שגיאה / תסמין הסיבה הפתרון
rejected (fetch first) ה-remote לא ריק בדוק שלא נוצר README ב-GitLab בטעות. אם כן — מחק את ה-repo וצור מחדש ריק.
remote: HTTP Basic: Access denied סיסמה/token שגויים צור Personal Access Token חדש ב-GitLab עם הרשאות write_repository.
Filename too long נתיבים ארוכים ב-Windows הרץ git config --global core.longpaths true והפעל מחדש את Git Bash.
Large file detected קובץ מעל 100MB מצא אותו (git ls-files | xargs ls -l | sort -k 5 -n | tail) ומחק או העבר ל-LFS.
ה-push מתקיים אבל ארוך מאוד ה-repo גדול (בגלל bin/obj שלא נוקו) בטל את ה-push, חזור לשלב הניקוי, והתחל מחדש.

עבודה שוטפת עם Git — המדריך המעשי

איך עובדים יום-יום עם Git, branches ו-commits מ-Visual Studio ומ-VS Code.

המודל המנטלי — ארבעת האזורים של Git

לפני שצוללים לפקודות, כדאי להבין איפה נמצאים הקבצים שלך בכל רגע. ב-Git יש ארבעה אזורים, וכל פעולה מעבירה קובץ ביניהם:

i
איך זה עובד בפועל?

אתה עורך קובץ ב-IDE (Working Directory) → עושה git add שמסמן אותו "מוכן ל-commit" (Staging) → git commit שמכניס אותו להיסטוריה המקומית (Local Repo) → git push שמשתף עם הצוות (Remote/GitLab).

אסטרטגיית Branches — איך מארגנים את העבודה

הבחירה הקלאסית למרבית הצוותים: GitLab Flow — פשוטה, גמישה, ומתאימה לעבודה גם עצמאית וגם בצוות:

הענפים — מי אחראי על מה?

  • main — מה שרץ ב-Production. לכל commit כאן צריך להגיע גרסה יציבה. לא נוגעים ישירות!
  • develop — הענף המרכזי לפיתוח השוטף. כל פיצ'ר חדש מתחיל ממנו וחוזר אליו.
  • feature/* — ענף לכל פיצ'ר חדש. נוצר מ-develop, מתמזג בחזרה אליו דרך Merge Request.
  • bugfix/* — לתיקון באגים שאינם דחופים. נוצר ומתמזג כמו feature.
  • hotfix/* — תיקון דחוף בפרודקשן. נוצר מ-main, חוזר ל-main וגם ל-develop.
  • release/* — להכנת גרסה לפני release רשמי (לא חובה לכל צוות).

Branch naming convention

hotfix/PROJ-310-critical-payment-error

מחזור עבודה יומי — Step by Step

זה התסריט שתחזור עליו פעמים רבות בכל יום. שמור אותו בראש:

1. בוקר טוב — עדכן את עצמך

לפני שמתחילים לעבוד, מושכים את העדכונים מהצוות: git checkout develop ואז git pull. זה מבטיח שאתה מתחיל מהבסיס העדכני ביותר.

2. צור branch לפיצ'ר חדש

לעולם לא עובדים ישירות על develop או main. צור branch חדש עם git checkout -b feature/PROJ-123-my-feature. השם צריך להיות תיאורי וכולל מספר טיקט אם יש.

3. עבוד, שמור, וחזור

עורך קוד ב-IDE כרגיל. שומר עם Ctrl+S. Git יזהה כל שינוי אוטומטית. אין צורך לעשות כלום עד שמגיעים לנקודת ביניים משמעותית.

4. בצע commits קטנים ותכופים

כל פעם שסיימת יחידת עבודה הגיונית (תיקון פונקציה, הוספת רכיב, השלמת test) — עשה commit. git add . ו-git commit -m "...". עדיף 5 commits קטנים מאחד גדול.

5. דחוף ל-GitLab

לפחות פעם ביום — git push כדי שהקוד שלך מגובה בשרת. אם זה ה-push הראשון של ה-branch: git push -u origin feature/...

6. סיום פיצ'ר → Merge Request

כשהפיצ'ר מוכן: ב-GitLab → Merge Requests → New. בחר source = ה-branch שלך, target = develop. הוסף תיאור, מבקש review, ולחץ Create.

7. אחרי merge — נקיון

כשה-MR מאושר ונסגר: git checkout develop → git pull → git branch -d feature/PROJ-123-my-feature (מוחק לוקלית). ב-GitLab בדרך כלל יש אופציה למחוק את הענף אוטומטית אחרי merge.

הפקודות שתשתמש בהן 95% מהזמן

git status
מציג את כל השינויים — אילו קבצים שונו, אילו ב-staging, אילו לא tracked. השתמש בה הרבה!
git pull
מושך את העדכונים האחרונים מ-GitLab למחשב. הפעולה הראשונה כל בוקר.
git checkout -b feature/X
יוצר branch חדש בשם X ועובר אליו. שילוב של git branch X + git checkout X.
git checkout develop
מחליף ל-branch קיים בשם develop. כל השינויים שלא commit-ed יישארו (אלא אם יש קונפליקט).
git add .
מוסיף את כל השינויים בתיקייה הנוכחית ל-staging. אפשר גם git add filename.cs לקובץ ספציפי.
git commit -m "msg"
יוצר commit עם ההודעה. ההודעה צריכה להיות תיאורית וקצרה (50 תווים בשורה הראשונה).
git push
דוחף את ה-commits החדשים שלך ל-GitLab. אחרי -u origin בפעם הראשונה, מספיק git push.
git log --oneline -10
מציג את 10 ה-commits האחרונים בשורה אחת לכל אחד. לסקירה מהירה.
git diff
מראה מה השתנה בקבצים שלא הוספת ל-staging עדיין. git diff --staged מראה את ה-staging.
git branch
רשימת כל ה-branches המקומיים. git branch -a כולל את ה-remote.
git stash
"שומר בצד" שינויים זמנית בלי לעשות commit. שימושי כשצריך פתאום לעבור branch באמצע עבודה.
git stash pop
מחזיר את מה ש-stashed בחזרה לעבודה.
git restore filename
מבטל שינויים בקובץ שעדיין לא ב-staging — מחזיר אותו למצב האחרון של commit.
git restore --staged f
מוציא קובץ מ-staging (חזרה ל-modified). לא מוחק את התוכן, רק את ה-add.

איך כותבים הודעת commit טובה?

הודעות commit הן תיעוד שיישאר לעד. עוד שנה תפתח את ה-log ותרצה להבין מה עשית. כללי זהב:

✓ נכון

Add user profile page with avatar upload

✗ שגוי

fixed stuff

✓ נכון

✗ שגוי

VS מאז 2019 בא עם Git מובנה ומצוין. הכלים העיקריים נמצאים תחת תפריט View:

  • Git Changes (Ctrl+0, Ctrl+G) — החלון המרכזי. מציג Modified files, Staged files, ושדה הודעת commit.
  • Git Repository — מציג graph של branches, history, ו-incoming/outgoing commits.
  • Branch picker — בפינה הימנית-תחתונה. לחיצה פותחת תפריט branch switching, יצירה, מחיקה.

תהליך עבודה מתוך VS

  1. Pull — בחלון Git Changes, לחץ על החץ ↓ ליד שם ה-branch.
  2. צור branch — לחץ על שם ה-branch בפינה תחתונה → New Branch → תן שם → Create.
  3. ערוך קוד — VS יסמן בצד שמאל של הקוד מה השתנה (קו צהוב = modified, ירוק = added).
  4. Stage + Commit — בחלון Git Changes לחץ + ליד קובץ ל-stage, או "Stage All". הקלד הודעה. לחץ Commit Staged.
  5. Push — לחץ על החץ ↑ או על "Push" בחלון Git Changes.
  6. Merge Request — VS יציע "Create Merge Request" — מקפיץ את הדפדפן ל-GitLab.
★
טיפ ל-VS 2022

הפעל את Tools → Options → Source Control → Git Global Settings וודא ש-Commit changes after merge by default מסומן. חוסך לחיצה.

VS Code — Source Control panel

ב-VS Code, Git מובנה ב-Activity Bar בצד שמאל (אייקון של branches). מקש קיצור: Ctrl+Shift+G.

  • Source Control panel — מציג את כל השינויים בחלוקה ל-Staged Changes ו-Changes.
  • Status Bar בתחתית — מציג את ה-branch הנוכחי, מספר הקבצים שהשתנו, ועדכונים זמינים. לחיצה פותחת תפריט branches.
  • Gutter indicators — בצד שמאל של כל קובץ פתוח מסומנים בצבעים השינויים: ירוק (חדש), כחול (שונה), אדום (נמחק).

תהליך עבודה מתוך VS Code

  1. Pull — לחץ על שם ה-branch ב-status bar → Pull. או Ctrl+Shift+P → "Git: Pull".
  2. צור branch — לחץ על שם ה-branch → Create new branch → הקלד שם.
  3. ערוך קוד — שמור עם Ctrl+S. סימוני שינויים יופיעו ב-Source Control.
  4. Stage — בחלון Source Control, רחף על שם הקובץ ולחץ +. או "Stage All Changes".
  5. Commit — הקלד הודעה בשדה למעלה, לחץ ✓ Commit (או Ctrl+Enter).
  6. Push — לחץ על "..." → Push. או על האייקון ↑ ב-status bar.

Extensions מומלצים

  • GitLens — מציג ליד כל שורה מי כתב אותה ומתי, היסטוריית קובץ, blame view, ו-graph מתקדם.
  • Git Graph — מציג graph ויזואלי של כל ה-commits וה-branches.
  • GitLab Workflow — אינטגרציה ישירה עם MRs, issues, ו-CI/CD pipelines של GitLab.
  • Conventional Commits — עוזר לכתוב הודעות commit בפורמט הנכון.

Git Bash / Terminal — הדרך הקלאסית

גם אם אתה משתמש ב-IDE, חשוב לדעת לעבוד מ-CLI. כשמשהו נשבר, ה-CLI נותן לך שליטה מלאה ולוגים מפורטים. לפעמים הוא גם פשוט מהיר יותר.

BASH · Daily workflow
# בוקר — עדכון
git checkout develop
git pull

# התחלת פיצ'ר חדש
git checkout -b feature/PROJ-123-add-export

# ... עובדים ב-IDE ...

# בדיקה לפני commit
git status
git diff

# commit ראשון
git add .
git commit -m "feat(reports): add Excel export base structure"

# ... ממשיכים לעבוד ועוד commits ...

git add src/services/export.service.ts
git commit -m "feat(reports): implement column mapping logic"

# push ראשון של ה-branch
git push -u origin feature/PROJ-123-add-export

# pushים הבאים — פשוט:
git push

# סיום היום — לוודא שהכול גובה
git status
git log --oneline -5

Aliases שימושיים — חוסכים זמן יקר

BASH · Git aliases
# הוסף לקובץ ~/.gitconfig או הרץ פעם אחת:

git config --global alias.s "status -sb"
git config --global alias.co "checkout"
git config --global alias.cb "checkout -b"
git config --global alias.br "branch"
git config --global alias.cm "commit -m"
git config --global alias.last "log -1 HEAD"
git config --global alias.lg "log --oneline --graph --decorate --all -20"
git config --global alias.unstage "restore --staged"
git config --global alias.amend "commit --amend --no-edit"

# עכשיו במקום: git status -sb → git s
# במקום: git checkout -b feature/x → git cb feature/x

תרחישים יומיומיים — איך פותרים?

🔄 צריך לעבור branch באמצע עבודה (יש שינויים לא-commit-ed)

BASH · Stash workflow
# שמור את השינויים זמנית
git stash save "WIP: working on user form validation"

# עבור ל-branch אחר ועשה מה שצריך
git checkout main
# ... do stuff ...

# חזור ל-branch המקורי ושחזר את השינויים
git checkout feature/my-work
git stash pop

# בונוס: רשימת כל ה-stashes
git stash list

↩️ ביצעתי commit עם הודעה שגויה

BASH · Amend commit
# רק אם עוד לא עשית push! משנה את ה-commit האחרון
git commit --amend -m "Correct commit message"

# אם רוצה גם להוסיף קובץ שכחת:
git add forgotten-file.cs
git commit --amend --no-edit

⚠️ עשיתי שינוי בטעות — איך מבטלים?

BASH · Undo changes
# בטל שינויים בקובץ ספציפי (שלא ב-staging)
git restore filename.cs

# בטל את כל השינויים בכל הקבצים
git restore .

# הוצא קובץ מ-staging (אם בטעות עשית git add)
git restore --staged filename.cs

# בטל את ה-commit האחרון אבל השאר את השינויים בקבצים (לא push-ed)
git reset --soft HEAD~1

# בטל את ה-commit האחרון ומחק את כל השינויים — זהירות!
git reset --hard HEAD~1
!
זהירות עם --hard ו-force push

git reset --hard מוחק שינויים לצמיתות. git push --force יכול למחוק עבודה של אחרים מה-remote. אם לא בטוח — תשאל לפני שתפעיל.

💥 התרחש Merge Conflict

BASH · Resolve conflict
# ראית את ההודעה: CONFLICT (content): Merge conflict in src/file.ts

# 1. בדוק אילו קבצים בקונפליקט
git status

# 2. פתח את הקבצים ב-IDE - תראה סימונים כמו:
#    <<<<<<< HEAD
#    הקוד שלך
#    =======
#    הקוד מה-branch השני
#    >>>>>>> develop

# 3. ערוך ידנית — מחק את הסימונים, השאר רק את הגרסה הסופית

# 4. סמן שהקונפליקט נפתר
git add src/file.ts

# 5. סיים את ה-merge
git commit
# (Git יציע הודעה אוטומטית — אפשר להשאיר)
i
VS Code פותר conflicts יפה

פתח את הקובץ ב-VS Code — מעל כל קונפליקט יופיעו כפתורים: Accept Current Change, Accept Incoming Change, Accept Both, או Compare Changes. לחיצה אחת ו-VS Code עושה את העריכה.

10 הכללים הזהובים

1. Pull לפני שמתחילים

תמיד התחל יום עבודה עם git pull על develop. חוסך מערבולת מאוחר יותר.

2. Branch לכל פיצ'ר

אף פעם לא ישירות על main/develop. branch ייעודי = שלוות נפש.

3. Commit קטנים ותכופים

5 commits קטנים > commit אחד גדול. קל יותר לסקירה ו-revert.

4. הודעות תיאוריות

"Fix login bug" עדיף על "fix". "Add export feature" עדיף על "wip".

5. Push לפחות פעם ביום

קוד שלא ב-GitLab — לא קיים. הדיסק שלך יכול להישבר היום.

6. אל תעשה commit ל-secrets

סיסמאות, API keys, certificates — לעולם לא ב-Git. רק ב-.env מקומי.

7. אל תעשה force push לענפים משותפים

הורס עבודה של אחרים. רק על branch אישי שלך, ובזהירות.

8. אל תעלה קבצים גדולים

סרטונים, מאגרי DB, dumps — חוץ מ-Git. השתמש ב-LFS או storage נפרד.

9. אל תערבב פיצ'רים ב-branch אחד

פיצ'ר חדש = branch חדש. תיקון באג = branch נפרד. קל יותר לעקוב ולהפוך.

10. בקש Code Review

גם אם אתה לבד — חזור על הקוד שלך לפני merge. עוד עיניים = פחות באגים.

09

צ'קליסט ההעברה

לחץ על כל פריט שהשלמת. מתעדכן בזיכרון הדפדפן.

לפני ההעברה

  • Git for Windows מותקן (git --version עובד ב-Git Bash)
  • הגדרות git הגלובליות בוצעו (user.name, user.email, autocrlf)
  • Personal Access Token או SSH Key הוגדרו ב-GitLab
  • Project ריק (ללא README) נוצר ב-GitLab
  • זוהתה גרסת Visual Studio של הפרויקט

במהלך ההעברה

  • הקוד הועתק מ-_sources ל-_staging
  • קבצי ClearCase נוקו (.keep, .contrib, lost+found)
  • Build artifacts נמחקו (bin, obj, .vs, packages)
  • גודל הפרויקט מתחת ל-1GB
  • .gitignore מתאים לגרסת VS נוצר בשורש
  • רשימת הקבצים ב-git status נראית תקינה
  • Commit ראשון עם הודעה תיאורית בוצע
  • Remote נוסף ו-push עלה בהצלחה

אחרי ההעברה

  • Fresh clone בוצע מתיקייה נפרדת
  • הפרויקט נפתח ב-VS ללא שגיאות
  • Build עובר בצורה נקייה (0 errors)
  • הפרויקט רץ כצפוי (F5)
  • Protected Branches הוגדרו ב-GitLab
  • README עם הנחיות build עודכן
  • הצוות קיבל את ה-clone URL החדש
  • ה-ClearCase view הישן סומן Read-Only / Archived
10

סיכום ותסריט מהיר

כל התהליך המלא, בבלוק אחד, לעתיד הקרוב שלך.

הסקריפט המלא — פרויקט יחיד

BASH · Full flow
#====== הגדרות — ערוך לפי הפרויקט ======
PROJECT_NAME="ProjectGamma"
VS_VERSION="2019"
GITLAB_URL="https://gitlab.com/your-group/project-gamma.git"

#====== 1. העתקה ל-staging ======
cd /c/Migration
cp -r "_sources/${PROJECT_NAME}-VS${VS_VERSION}" "_staging/${PROJECT_NAME}"
cd "_staging/${PROJECT_NAME}"

#====== 2. ניקוי ClearCase ======
find . -type d -name "lost+found" -exec rm -rf {} +
find . -name "*.keep*" -delete
find . -name "*.contrib*" -delete
find . -name "view.dat" -delete

#====== 3. ניקוי Build artifacts ======
find . -type d \\( -name "bin" -o -name "obj" -o -name ".vs" \
  -o -name "packages" -o -name "node_modules" \\) -exec rm -rf {} +
find . -name "*.suo" -delete
find . -name "*.user" -delete

#====== 4. הצבת .gitignore ======
cp "../../_gitignore-templates/gitignore-vs${VS_VERSION}.txt" .gitignore

#====== 5. Init + Commit + Push ======
git init
git branch -M main
git add .
git status   # בדיקה אחרונה!
git commit -m "Initial import from ClearCase (VS${VS_VERSION})"
git remote add origin "$GITLAB_URL"
git push -u origin main

echo "✓ Migration complete: $PROJECT_NAME"
i
עצה אחרונה

אל תריץ את הסקריפט הזה אוטומטית בלי git status באמצע. ההעברה הראשונה של כל פרויקט דורשת עין אנושית. אחרי שהצלחת 2-3 פרויקטים — מרגישים את הקצב, ואפשר לאוטומט.
